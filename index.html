1 · Pre-reqs — Licensing, Roles & Connectors
What license does the customer need to migrate to Purview DLP? ▶

Required: Microsoft 365 E5 or Microsoft 365 E5 Compliance (or equivalent standalone Purview DLP license).

3P apps (Box, Dropbox, Google Workspace, Salesforce): Require PAYG (Pay-As-You-Go) billing configured in addition to the base license.

What to tell the customer: Check Microsoft 365 admin center → Billing → Licenses. If E5 Compliance is missing, they need to purchase before migration can start.

Do MDA customers need to purchase additional licenses to migrate to Purview DLP? ▶

No — MDA customers are NOT required to purchase any additional licenses. Microsoft is providing a promotional SKU (100% discount) to all MDA licensed customers:

ScenarioWhat the customer needsSetup required?Promo coverage
1P policies (SharePoint, OneDrive, Teams) IP&G licensing. Customers without IP&G will receive the promo SKU. No setup needed — works as-is once IP&G is confirmed. 100% discount via promo SKU
3P policies (Box, Dropbox, Google Workspace, Salesforce) IP&G + 3P Pay-As-You-Go (PAYG) account + connectors configured. Setup required — customer must configure a PAYG billing account in Azure AND turn on 3P connectors before 3P DLP capabilities work in Purview. 100% discount via promo SKU (covers PAYG charges)

What to tell the customer: There is no net new cost for migrating from MDA DLP to Purview DLP. The promo SKU covers both IP&G entitlement and 3P PAYG charges for all MDA licensed customers. For 3P customers, remind them that PAYG setup and connector configuration are required steps before policies go live. For further licensing questions, loop in your CSAM or Business Planning contact.

What admin roles are needed to migrate to Purview? ▶

Two roles required — both must pass:

PortalRequired Role
MDA (security.microsoft.com)Global Administrator or Security Administrator
Purview (purview.microsoft.com)Compliance Administrator and/or Information Protection Administrator

Both roles are required. Admin must fix any missing role assignment before migration can proceed.

Do I need to set up connectors before migrating? ▶

1P (SPO/ODB/Exchange): No connector setup required — SPO, OneDrive, and Exchange are native Purview locations.

3P (Box, Dropbox, Google Workspace, Salesforce): MDA app connectors must be active before Purview 3P DLP can work.

Already connected in MDA? No action needed — existing connectors carry over. Customer just needs to configure the Purview DLP policy targeting that instance.

Not yet connected? Must connect in MDA first: security.microsoft.com → Cloud Apps → Connected apps → App connectors.

What is PAYG and why is it needed for 3P policies? ▶

PAYG = Pay-As-You-Go — a billing model for Purview capabilities applied to third-party connected apps (Box, Dropbox, Google Workspace, Salesforce). Running DLP on these apps incurs metered usage charges beyond the base license.

What the customer must do: Configure PAYG billing in Azure before using 3P DLP in Purview.

Best practice: Should I enforce Purview policies immediately after creating them? ▶

No — always run in Simulation mode first. This applies to both 1P DLP policies and auto-labeling policies.

Policy typeRecommended first stepWhen to enforce
1P DLP policy (SPO/ODB/Exchange)Set to Simulation mode — logs matches, no enforcementAfter 1–2 weeks of validated coverage and tuned false positives
Auto-labeling policyRun simulation — review which items would be labeledAfter confirming correct items match and label is published to users
3P DLP policy (Box, Dropbox etc.)Created as Disabled — review before turning onAfter confirming connector is healthy and PAYG is configured

To enable enforcement: purview.microsoft.com → DLP → Policies → [Policy] → Edit → Turn it on
For auto-labeling: purview.microsoft.com → Information Protection → Auto-labeling → [Policy] → Turn on

3 · 3P Policies — Box, Dropbox, Google Workspace, Salesforce
My MDA policy monitored Box / Dropbox / Google Workspace / Salesforce. Does Purview support these? ▶

Yes — these four apps are supported in Purview DLP via the "Instances" location. AWS, ServiceNow, and Cisco Webex are not supported (no Purview equivalent).

Pre-req: MDA app connector must be active. If already connected in MDA, no re-setup needed — connector carries over automatically to Purview Instances.

4 · Auto-labeling Policies
My MDA policy applied a sensitivity label. Why do I need a separate Purview policy for that? ▶

What changed: In MDA, "Apply sensitivity label" was a governance action inside the file policy. In Purview, label application and DLP enforcement are separate policy types.

  • Purview DLP policy → detects content and restricts access / alerts
  • Purview auto-labeling policy (Information Protection) → applies the sensitivity label

When migrating an MDA policy that applied a label, two Purview policies are needed — both created in Simulation mode.

Navigation: purview.microsoft.com → Information Protection → Auto-labeling

Does auto-labeling work for 3P apps (Box, Dropbox)? ▶

Not yet. Auto-labeling in Purview Information Protection currently supports first-party workloads only (SharePoint, OneDrive, Exchange). 3P app auto-labeling is on the roadmap but not yet available.

If a customer's MDA policy applied labels to Box/Dropbox files, this is a confirmed gap. Do not promise a date — confirm with PG before advising.

5 · Already in Purview — Post-migration questions
My Purview DLP policy is enabled but it's not blocking anything. What's wrong? ▶

This is a Purview CSS question — hand off the case. Once a policy is created and enabled in Purview, enforcement issues are owned by Purview CSS.

Before handing off, verify:

  • Policy is not still in Simulation mode (check policy status in Purview DLP)
  • For 3P: MDA connector is healthy — check in security.microsoft.com → Cloud Apps → Connected apps
  • Policy is scoped to the correct location (sites, users, or app instance)

If connector is healthy and policy is on but still not enforcing → route to Purview CSS.

→ See: Escalation Guide for routing steps

Labels are not being applied by my auto-labeling policy. What should I check? ▶

This is a Purview CSS question — hand off if basic checks pass.

Check first (MDA CSS scope):

  • Is the auto-labeling policy still in Simulation mode? If yes — labels won't apply until turned on.
  • Is the sensitivity label published to the relevant users in the label policy?
  • Is the policy scoped to the correct SharePoint site / OneDrive accounts?

If all checks pass and labels still not applying → route to Purview CSS.

Can MDA and Purview policies run at the same time? What happens if both are active? ▶

No — they cannot co-exist. Running equivalent policies in both MDA and Purview simultaneously creates enforcement conflicts: files may be double-matched, actions may conflict, and alerts will be duplicated.

Correct sequence once satisfied with Purview coverage:

  1. Run Purview policy in simulation for 1–2 weeks — validate coverage, tune false positives
  2. Disable the corresponding MDA policy — security.microsoft.com → Cloud Apps → Policies → Edit → Disable
  3. Enable the Purview policy for enforcement
  4. Monitor for 1–2 weeks, then delete the MDA policy
Never enable a Purview policy while the equivalent MDA policy is still active.

Always Capture Before Routing

Regardless of stage — gather this before handing off any case.

  • Tenant ID + org name — required for any handoff
  • Migration stage — where is the customer? (Assessing / Building policies / Testing in Simulation / Rolling out enforcement / Decommissioning MDA)
  • Issue description — exact capability or behavior, confirmed / reproduced
  • Gap vs. break confirmed — checked against Parity Depth Library
  • MDA policy status — is the MDA policy still active? (co-existence risk)
  • Business impact — what protection is at risk if unresolved
  • Steps already taken — what was tried and what the result was
  • Urgency signal — churn risk, exec visibility, Dec 31 2026 deadline pressure

Glossary
Support Layer Owner: CSS DLP Migration Team Last updated: May 2026
MDA
Microsoft Defender for Cloud Apps — cloud security product that includes file policies for detecting and governing sensitive content in cloud apps. File policies are being deprecated Dec 31, 2026.
Purview DLP
Microsoft Purview Data Loss Prevention — the target platform for migrated file policies. Supports SPO, ODB, Exchange, Teams, Endpoints, and 3P apps via MDA connectors.
File Policy
An MDA construct that detects files matching filters (content, metadata, sharing) and applies governance actions. Being replaced by Purview DLP policies.
DLP Rule
A rule inside a Purview DLP policy. One policy can contain multiple rules, each with its own conditions and actions.
Sensitive Info Type (SIT)
A classifier in Purview that detects specific sensitive content patterns (e.g., Credit Card Number, SSN). Equivalent to MDA preset expressions / DCS content inspection.
Sensitivity Label
A classification tag (e.g., Confidential, Public) applied to files or emails. In Purview, applied via auto-labeling policy — not inside a DLP policy.
Auto-Labeling Policy
A separate Purview Information Protection policy that applies sensitivity labels based on content. Required when migrating MDA file policies that had "apply label" as a governance action.
Simulation Mode
A Purview DLP policy state where policies log matches without enforcing. Run for 1–2 weeks before turning on enforcement. Not available in MDA.
Admin Quarantine
A DLP action that moves a file to an admin-controlled quarantine folder. Supported in both MDA and Purview.
User Quarantine
An MDA action that moves a file to a user's personal quarantine folder. No direct equivalent in Purview — workaround: Restrict access + Power Automate.
Instances (location)
A Purview DLP location type for third-party connected apps (Box, Google Workspace, Dropbox, Salesforce). Requires an active MDA connector.
Parity Gap
A capability present in MDA that has no current equivalent in Purview. Examples: remove specific collaborator, folder-level scoping, AWS support.
Parity (Full / Partial / Gap / Roadmap)
Status labels used in the Parity Depth Library. Full = direct equivalent exists. Partial = workaround available. Gap = no equivalent. Roadmap = planned but not yet available.
CSS
Customer Success / Support — the team this toolkit is built for. MDA CSS handles migration clarity; Purview CSS handles Purview-side issues post-migration.
Power Automate
Microsoft's workflow automation tool. Used as a workaround for MDA governance actions with no Purview DLP equivalent (trash, user quarantine, remove collaborator).
Activity Explorer
A Purview tool showing DLP events across workloads. Equivalent to where customers previously monitored MDA file policy matches.
Deprecation deadline
December 31, 2026 — the date MDA file policies are turned off. Customers must complete migration before this date.
DSP
Data Security Posture — used in internal program naming (MDA Convergence Zen - DSP Only).
Cohort
A customer tier grouping used in migration wave planning. Cohort 1 typically = highest priority / most complex customers.
S500
Strategic 500 — Microsoft's top strategic accounts. Require escalated handling — do not handle solo.
TCO
Total Cost of Ownership — used when customers raise cost as an objection to migration. Commercial conversation owned by account team, not CSS.

Warm Handoff Protocol

StepActionOwner
1Confirm customer is actively migrating or has completed migration to Purview DLPMDA CSS engineer
2Check parity gap — if issue is a known Cannot Migrate condition, route to product feedback intake (do not hand to Purview CSS)MDA CSS engineer
3If issue is Purview-side (config, policy behaviour, connector), initiate warm handoff to Purview CSS SPOCMDA CSS engineer
4Purview CSS SPOC acknowledges and picks up the case — customer is introduced to the new engineerPurview CSS SPOC
5MDA CSS engineer remains CC'd until customer confirms resolutionMDA CSS engineer

What NOT to hand off

  • Cases where the root cause is a parity gap (Cannot Migrate condition) — route to product feedback intake instead
  • Cases still in pre-migration assessment — keep with MDA CSS until customer is ready to migrate
  • Licensing questions — handle via CSAM or Business Planning contact, not Purview CSS
