1 · Pre-reqs — Licensing, Roles & Connectors
What license does the customer need to migrate to Purview DLP? ▶

Required: Microsoft 365 E5 or Microsoft 365 E5 Compliance (or equivalent standalone Purview DLP license). The migration tool validates this at the Licensing Gate — if it fails, the customer cannot proceed.

3P apps (Box, Dropbox, Google Workspace, Salesforce): Require PAYG (Pay-As-You-Go) billing configured in addition to the base license. The tool surfaces a PAYG acknowledgement checkbox before creating 3P policies.

What to tell the customer: Check Microsoft 365 admin center → Billing → Licenses. If E5 Compliance is missing, they need to purchase before migration can start.

→ Open question OQ5: IP&G licensing per-user vs tenant-level — confirm with Pallav before advising.

What admin roles are needed to run the migration tool? ▶

Two roles required — both must pass:

PortalRequired Role
MDA (security.microsoft.com)Global Administrator or Security Administrator
Purview (purview.microsoft.com)Compliance Administrator and/or Information Protection Administrator

The Permission Gate checks both. If either fails, the tool surfaces the specific missing role with a docs link. Admin must fix the role assignment before progressing.

→ Open questions OQ2 (MDA role) and OQ3 (Purview role) — confirm exact role names with Eng/Srivalli before advising.

Do I need to set up connectors before migrating? What is the O365 connector? ▶

1P (SPO/ODB/Exchange): The O365 connector in MDA may be required for SPO/ODB migration pre-req. Status: open question OQ4 — confirm with Pallav before advising customers this is mandatory.

3P (Box, Dropbox, Google Workspace, Salesforce): MDA app connectors must be active before Purview 3P DLP can work. The migration tool checks connector status in the Prerequisites card.

Already connected in MDA? No action needed — existing connectors carry over. Customer just needs to configure the Purview DLP policy targeting that instance.

Not yet connected? Must connect in MDA first: security.microsoft.com → Cloud Apps → Connected apps → App connectors.

The "Migrate to Purview" button is blocked / not available. Why? ▶

The tool runs two gates before showing any policies. Check which gate is failing:

SymptomCauseFix
Licensing check failsNo active E5 / E5 Compliance licensePurchase / assign license. Tool blocked until resolved.
Permission check failsMissing MDA or Purview admin roleAssign correct role in respective admin center. Both must pass.
Gov cloud tenantGov cloud tenants are blocked in v1Not supported in v1 — no workaround. Document as Cannot Migrate.
What is PAYG and why is it needed for 3P policies? ▶

PAYG = Pay-As-You-Go — a billing model for Purview capabilities applied to third-party connected apps (Box, Dropbox, Google Workspace, Salesforce). Running DLP on these apps incurs metered usage charges beyond the base license.

What the customer must do: Configure PAYG billing in Azure before using 3P DLP in Purview. The migration tool shows an inline PAYG acknowledgement checkbox — Create button is disabled until checked.

→ Open question OQ6: PAYG meter — customer-configured or auto-provisioned? Confirm with Billing before advising setup steps.

Why is my migrated SPO/ODB policy in Simulation mode and not enforcing? ▶

By design. The migration tool creates all SPO/ODB policies in Simulation mode ("Test it out first") — policies log matches without any enforcement action. This is a deliberate draft-first approach to prevent accidental enforcement.

What to do: Run in simulation for 1–2 weeks, review matches in Activity Explorer, compare against MDA match volumes, tune false positives, then manually enable enforcement.

To enable enforcement: purview.microsoft.com → DLP → Policies → [Policy] → Edit → Change mode to "Turn it on".

3 · 3P Policies — Box, Dropbox, Google Workspace, Salesforce
My MDA policy monitored Box / Dropbox / Google Workspace / Salesforce. Does Purview support these? ▶

Yes — these four apps are supported in Purview DLP via the "Instances" location. AWS, ServiceNow, and Cisco Webex are not supported (no Purview equivalent).

Pre-req: MDA app connector must be active. If already connected in MDA, no re-setup needed — connector carries over automatically to Purview Instances.

Policy split warning: If a single MDA policy has both TrashFile AND ContentExtensionMatchesWords, the tool creates two separate Purview policies [D] and [E]. Admin warned: same file may match both — behavior differs from the original single MDA policy.

Why are my 3P policies created as Disabled (not Simulation like 1P)? ▶

By design. 3P app DLP policies are created in Disabled mode (policy off) — not Simulation. This is because PAYG billing is involved: even simulation mode for 3P apps can incur metered charges. Admin must explicitly review and enable each 3P policy.

To enable: purview.microsoft.com → DLP → Policies → [Policy] → Edit → Turn on. Ensure PAYG is configured and connector is healthy before enabling.

4 · Auto-labeling Policies
My MDA policy applied a sensitivity label. Why do I need a separate Purview policy for that? ▶

What changed: In MDA, "Apply sensitivity label" was a governance action inside the file policy. In Purview, label application and DLP enforcement are separate policy types.

  • Purview DLP policy → detects content and restricts access / alerts
  • Purview auto-labeling policy (Information Protection) → applies the sensitivity label

The migration tool creates both automatically from a single MDA policy that had a label action. Both are created in Simulation mode.

Navigation: purview.microsoft.com → Information Protection → Auto-labeling

My auto-labeling policy is in Simulation — when will labels actually apply? ▶

Simulation mode logs which items would get labeled without actually applying labels. This is a safe review step.

To review simulation results: purview.microsoft.com → Information Protection → Auto-labeling → [Policy] → Items to review

To turn on enforcement: After validating simulation results, enable the policy. Labels will apply to new and existing content going forward.

Pre-req check: Verify the sensitivity label is published to the relevant users before enabling — unpublished labels will not apply even when the policy is on.

Does auto-labeling work for 3P apps (Box, Dropbox)? ▶

Not yet. Auto-labeling in Purview Information Protection currently supports first-party workloads only (SharePoint, OneDrive, Exchange). 3P app auto-labeling is on the roadmap but not available in v1.

If a customer's MDA policy applied labels to Box/Dropbox files, this is a confirmed gap. Do not promise a date — confirm with PG before advising.

5 · Policy Classification — Can / Partial / Cannot Migrate
What does Can Migrate / Partial Migrate / Cannot Migrate mean? ▶
StatusMeaningAdmin action
✅ Can MigrateAll conditions, filters, and actions have Purview equivalentsNone — proceeds automatically
⚠️ Partial MigrateSupported subset migrates; unsupported elements flagged with reason codesMust explicitly acknowledge each gap before proceeding
❌ Cannot MigrateNo supported conditions or actions — policy not sent to PurviewReview reason code. Plan manual recreation or accept gap.

Classification is based purely on filter/condition/action Purview equivalence — not on licensing, PAYG status, or admin permissions.

My policy shows Cannot Migrate — what are common reasons? ▶
Reason code / scenarioWhyWhat to do
ContentIsNotLabeled conditionNo Purview DLP equivalent for "file has no label" detectionUse auto-labeling to label files, then build DLP on labeled content
Target app = AWS / ServiceNow / WebexNot supported in Purview DLPEvaluate native DLP in those platforms. Confirmed gap.
No supported conditions or actions at allEntire policy uses unsupported elementsManual recreation needed or accept gap. Document for PG feedback.
My policy is Partial Migrate — what gets migrated and what doesn't? ▶

The supported subset of conditions and actions migrates. Unsupported elements are:

  • Flagged with reason codes in the readiness table
  • Written to the Purview draft policy's description field as workaround notes
  • Visible to the admin — not hidden

Admin must explicitly acknowledge each gap before the tool proceeds. Common partial scenarios:

  • MIME type filter → maps to file extensions only (information loss possible)
  • Sharing level "Private" → skipped (no Purview equivalent)
  • Parent folder condition → in development (ETA 6/27 — do not promise)
  • 3P policy with TrashFile + ContentExtensionMatchesWords → creates two policies [D]+[E]
6 · Already in Purview — Post-migration questions
My Purview DLP policy is enabled but it's not blocking anything. What's wrong? ▶

This is a Purview CSS question — hand off the case. Once a policy is created and enabled in Purview, enforcement issues are owned by Purview CSS.

Before handing off, verify:

  • Policy is not still in Simulation mode (check policy status in Purview DLP)
  • For 3P: MDA connector is healthy — check in security.microsoft.com → Cloud Apps → Connected apps
  • Policy is scoped to the correct location (sites, users, or app instance)

If connector is healthy and policy is on but still not enforcing → route to Purview CSS.

→ See: Escalation Guide for routing steps

Labels are not being applied by my auto-labeling policy. What should I check? ▶

This is a Purview CSS question — hand off if basic checks pass.

Check first (MDA CSS scope):

  • Is the auto-labeling policy still in Simulation mode? If yes — labels won't apply until turned on.
  • Is the sensitivity label published to the relevant users in the label policy?
  • Is the policy scoped to the correct SharePoint site / OneDrive accounts?

If all checks pass and labels still not applying → route to Purview CSS.

I see a co-existence warning in Purview. What does it mean and when does it go away? ▶

The co-existence banner in Purview is shown when migrated policies exist as drafts. It persists until the admin manually dismisses it.

What the banner means: "You have policies migrated from MDA. Disable MDA policies before enabling Purview policies to avoid enforcement conflicts."

Correct sequence:

  1. Review Purview draft policies in simulation — validate coverage
  2. Disable MDA file policies (security.microsoft.com → Cloud Apps → Policies → Edit → Disable)
  3. Enable Purview policies one at a time — monitor for 1–2 weeks
  4. Delete MDA policies only after Purview is confirmed working
  5. Dismiss the banner
My MDA policy still shows "Not Migrated". Why wasn't it migrated? ▶
MDA Policy StateReason
Not MigratedNever attempted, classified as Cannot Migrate, or a mid-split rollback occurred
MigratedAll applicable Purview policies created successfully
Partially MigratedSupported subset created; gaps documented in Purview draft description

If mid-split rollback occurred (partial creation failed), the tool rolls back fully — no orphan policies in Purview. Admin can re-run the tool safely (idempotent re-run, no duplicates).

⚡ Escalation & Routing — Where does this go?

🔴 Something is broken
GA behavior that worked before is no longer working — DLP not enforcing, labels applying incorrectly, data missing from Content Explorer.
→ Route to Purview CSS (hand off case)
MDA CSS confirms it's a Purview-side break, then transfers. You do not open the ticket — you hand off.
🟡 Feature gap / customer wants more
Customer asks for a capability Purview doesn't have yet — remove specific collaborator, folder scoping, AWS app support, user quarantine.
→ Product feedback (NOT a support ticket)
Log via Purview DLP Product Team field feedback intake. Acknowledge the gap clearly to the customer — do not promise a date.

Routing Decision Framework

❓ Is GA behavior broken or regressed?
Yes → Route to Purview CSS — hand off the case. Confirm it's a Purview-side break first. No → Continue ↓
❓ Is this a confirmed parity gap (capability in MDA, absent in Purview today)?
Yes → Product feedback — log via Purview DLP Product Team field feedback intake. Do NOT open a support ticket. Acknowledge the gap honestly to the customer. No → Continue ↓
❓ Is this a 3P connector issue (Box, Dropbox, Google Workspace, Salesforce not connecting)?
Yes → MDA CSS owns — connector setup and auth issues stay with MDA CSS regardless of migration status. No → Continue ↓
❓ Is the customer asking a migration guidance question (how to, what maps to what)?
Yes → MDA CSS handles — use FAQ block, Parity Depth Library, What Stays / What Changes. No → Escalate to manager for triage.

📋 Evidence Checklist — capture before routing any escalation

  • Customer identity — Tenant ID, org name, segment, cohort
  • Issue description — exact capability or behavior, confirmed / reproduced
  • Gap vs. break confirmed — checked against Parity Depth Library. Is this a known gap or unexpected regression?
  • Connector status (3P only) — is the MDA connector live and healthy? Check before routing to Purview CSS.
  • Business impact — what breaks or is at risk if unresolved
  • Steps already taken — what CSS tried and what the result was
  • Urgency signal — churn flag, exec involvement, compliance deadline pressure
  • Migration phase — where is the customer in their migration? (Assessing / Building / Testing / Rolling out / Decommissioning)

Glossary
💬 Support Layer Owner: CSS DLP Migration Team Last updated: May 2026
MDA
Microsoft Defender for Cloud Apps — cloud security product that includes file policies for detecting and governing sensitive content in cloud apps. File policies are being deprecated Dec 31, 2026.
Purview DLP
Microsoft Purview Data Loss Prevention — the target platform for migrated file policies. Supports SPO, ODB, Exchange, Teams, Endpoints, and 3P apps via MDA connectors.
File Policy
An MDA construct that detects files matching filters (content, metadata, sharing) and applies governance actions. Being replaced by Purview DLP policies.
DLP Rule
A rule inside a Purview DLP policy. One policy can contain multiple rules, each with its own conditions and actions.
Sensitive Info Type (SIT)
A classifier in Purview that detects specific sensitive content patterns (e.g., Credit Card Number, SSN). Equivalent to MDA preset expressions / DCS content inspection.
Sensitivity Label
A classification tag (e.g., Confidential, Public) applied to files or emails. In Purview, applied via auto-labeling policy — not inside a DLP policy.
Auto-Labeling Policy
A separate Purview Information Protection policy that applies sensitivity labels based on content. Required when migrating MDA file policies that had "apply label" as a governance action.
Simulation Mode
A Purview DLP policy state where policies log matches without enforcing. Run for 1–2 weeks before turning on enforcement. Not available in MDA.
Admin Quarantine
A DLP action that moves a file to an admin-controlled quarantine folder. Supported in both MDA and Purview.
User Quarantine
An MDA action that moves a file to a user's personal quarantine folder. No direct equivalent in Purview — workaround: Restrict access + Power Automate.
Instances (location)
A Purview DLP location type for third-party connected apps (Box, Google Workspace, Dropbox, Salesforce). Requires an active MDA connector.
Parity Gap
A capability present in MDA that has no current equivalent in Purview. Examples: remove specific collaborator, folder-level scoping, AWS support.
Parity (Full / Partial / Gap / Roadmap)
Status labels used in the Parity Depth Library. Full = direct equivalent exists. Partial = workaround available. Gap = no equivalent. Roadmap = planned but not yet available.
CSS
Customer Success / Support — the team this toolkit is built for. MDA CSS handles migration clarity; Purview CSS handles Purview-side issues post-migration.
Power Automate
Microsoft's workflow automation tool. Used as a workaround for MDA governance actions with no Purview DLP equivalent (trash, user quarantine, remove collaborator).
Activity Explorer
A Purview tool showing DLP events across workloads. Equivalent to where customers previously monitored MDA file policy matches.
Deprecation deadline
December 31, 2026 — the date MDA file policies are turned off. Customers must complete migration before this date.
DSP
Data Security Posture — used in internal program naming (MDA Convergence Zen - DSP Only).
Cohort
A customer tier grouping used in migration wave planning. Cohort 1 typically = highest priority / most complex customers.
S500
Strategic 500 — Microsoft's top strategic accounts. Require escalated handling — do not handle solo.
TCO
Total Cost of Ownership — used when customers raise cost as an objection to migration. Commercial conversation owned by account team, not CSS.
